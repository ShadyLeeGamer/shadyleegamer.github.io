Edge Runner

Recently published a high-speed anime platformer on Steam, with the help of my sister (artist) for designing and animating the player and levels. The game features complex gravity-bending physics with open levels and mechanics that encourages fast-paced gameplay.

The biggest technical challenge for me was developing the raycast and engine physics character controller which functions on a variety of mechanics during gameplay.

Edge Runner

Raycast & Engine Physics Character Controller

Collision

Collision is handled by raycasting a circle collider onto surfaces.

This method was inspired by the use of spheres as hitboxes in Super Mario Galaxy, due to its round shape, we can prevent the side effects of gravity bending physics.

I found this to be more suitable than sensor rays as used in the classic Sonic games due to the bugs that occur when standing on slanted ledges and pointy terrain. This is demonstrated in the clip below of the WIP classic Sonic physics prototype of I was making.

Edge Running

While wall running, when the player reaches the edge of a wall, they will perform an ‘edge run’. where they drift off at a constant velocity. This stunt can also be used to seamlessly cross over small gaps.

Edge run is executed when there is no surface underneath the player or a steep slope angle threshold has been reached.

This is made possible by the CONTROL LOCK DISTANCE mechanic.

Wall jumping & Ricocheting

Wall jump is achieved by reflecting the player’s velocity off the surface as well as initiating a CONTROL LOCK.

The reflected velocity is then normalised with some of the surface normal to bounce further away from the wall. This adjustment was made to more easily wall jump over spikes while wall running more effectively.

One secret mechanic I really wanted to include was the ability to ricochet in tunnels. Like external forces, doing so allows you to gain instant momentum by exceeding the normal speed capacity.

However, this did not work well with the adjustment added as it causes the player to eventually ricochet in place.

To overcome this, I excluded the normal vector if the player was ricocheting, which is defined by if they were constantly wall jumping without spamming the input and without unlocking the control lock distance. This was the result:

Control Lock Distance

The control lock distance is used to prevents directional input from influencing the player’s speed until they reach a certain amount of distance from lock point.

This system was made flexible for use and is what ties all the other mechanics together such as edge running, wall jumping and hitting a launch pad.

In older solutions, the control lock timer concept was used, however it wasn’t suitable as the amount of distance covered during control lock was inconsistent due to speed always changing, which conflicted level design.

Assist System (inspired by Celeste)

The game aids player inputs to better achieve desired actions through assists like snapping to missed nearby environments, dash distance extension, input buffering, etc.

Momentum-based movement

The player’s momentum is influenced by activity like player mechanics (running along slopes, edge running, wall jumping) and external forces (launch pads, and moving or rotating platforms).

Pushing Splines to Limits!

With the world being built entirely out of splines, I developed an extension for Unity's Sprite Shape API that breaks splines down to segments and points for full control and flexibility.

It is used for entity movement movement on splines like player grinding on rails, hazards moving on walls.

It is also used to aid level design by displaying paths and point within a spline.

Here we can see the script being used to break down the grind rail spline into evenly spaced points to allow the player to move along the rail.

It is used similarly here for hazards moving along a path inside a wall. The spline path and points are visualised with an additional custom level editor tool.
